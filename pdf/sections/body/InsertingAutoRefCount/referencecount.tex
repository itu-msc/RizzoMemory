\section{Automatically reference counting Rizzo}
\label{section:refcount}

% \todo{Quick refresh of assumptions from \textit{Reference Counting section}}

The algorithms by \citeauthor{ullrich_counting_2021}~\cite{ullrich_counting_2021} are defined over the calculi $\lambda_{pure}$ and $\lambda_{RC}$, where the $\lambda_{RC}$ extends $\lambda_{pure}$ with reference counting instructions Figure~\ref{figure:refcount_IR} shows the IR used in our compiler, where Rizzo-specific extensions are highlighted in green.  

In this IR, $\texttt{pap}\; c \; \bar p$ denotes partial application of a global constant $c$, $\texttt{ctor}_i \; \bar p$ constructs a value of some erased type with tag $i$, and $\texttt{proj}_i \; x$ projects the $i\text{-th}$ field from constructor value $x$. Fully applied calls and partial applications may only happen on global constants and variable application is defined separately for only a single argument.

  p &\in& \textcolor{greenadditions}{Primitive \;::=\; x \mid l}\\[2pt]
  e &\in&
  \begin{array}[t]{@{}l@{}}
  Expr \;::=\; c\; \textcolor{greenadditions}{\bar p}
  \mid \mathbf{pap}\; c\; \textcolor{greenadditions}{\bar p}

Observe that the IR is defined in A-normal form~\cite{flanagan_essence_1993}, where expressions are non-recursive and are only allowed on the right-hand-side of let-bindings. 
Our representation keeps this shape but introduces \textit{Primitives} so that variables and literals (i.e.~integers, booleans, strings, and \texttt{never}) can appear directly in returns and as arguments in function/constructor application. This makes translation easier and avoids bindings such as $\texttt{let} \; x = 20; \; \texttt{ret} \; x$ when $\texttt{ret}\;20$ is enough.
We additionally add a specialised $\texttt{signal}$ constructor that can be used in places where $\texttt{ctor}_i$ is expected. 
This allows runtime to know the fresh memory cell should be stored in the signal heap (see Section~\ref{section:frp:signal_heap}). 
Finally, we add top level values as mappings from constants to $FnBody$ where the $FnBody$ is run at program start up to initialise the value. This way, we can reuse the automatic reference counting of Section~\ref{section:refcount:beans}.

\subsection{Transformations}\label{section:refcount:transforms}

%Goal of section: pretty little flow chart showing how we chain transformations together and an explanation of why/what in relation the assumptions}
To apply automatic reference counting, we first transform the source program into a shape that satisfies the invariants from Section~\ref{sec:background:beans:assumptions}, then lower it to the reference counting IR, and finally run the three algorithms $R$, $collect_{\mathbb{O}}$, and $C$~\cite{ullrich_counting_2021}.

Figure~\ref{figure:refcount:transformations} shows the transformation pipeline. Most transformations are standard compiler steps, but each contributes to the invariants required by the RC insertion algorithms.

\begin{figure}[h]
	\centering
	\includegraphics[width=\textwidth]{flowchart_transformations.png}
	\caption{Compiler passes that a Rizzo program goes through when running the compiler. Transformations inside the green (upper) box receive Rizzo programs as input and in the blue (lower) box they operate on the RC IR.}\label{figure:refcount:transformations}
\end{figure}

The compiler performs consecutive lambda elimination, lambda lifting, copy propagation, alpha-renaming, pattern lowering, dead let elimination, and A-normalization. Consecutive lambda elimination is not strictly required but reduces the number of lambdas to be lifted. 

The lowering of patterns into let-bindings may not directly contribute to any invariant, however it is required since there is no pattern matching in the IR.\@ The result of translating a match such as \lstinline|match mysig with x :: xs -> e| is shown in Listing~\ref{listing:refcount:elim_pattern}. The match expression must be kept around as it is the only point where reset/reuse pairs can be inserted, we will discuss this further in Section~\ref{section:refcount:beans}.

\begin{lstlisting}[caption={Output of lowering a pattern match on a signal value}, label={listing:refcount:elim_pattern}]
match mysig with
| x :: xs -> 
  let x = head mysig in 
  let xs = tail mysig in e
\end{lstlisting}

While some of the invariants are clearly fulfilled by the transformation pipeline, such as the requirement that all lambdas be lifted or that trivial bindings like \lstinline|let x = y| have been eliminated, some of them are not so obvious. 
In particular, the ones about application shape. The RC IR assumes unary variable applications and that applications on global constants are never over-applied. 
Therefore, when \textit{converting to IR} variable applications are rewritten such that \lstinline|x y z| becomes \lstinline|(x y) z| and over-applied global constants are rewritten to a call followed by a series of unary variable applications.

Another non-obvious point is the \textit{eta-expansion} of constructors. This invariant is enforced by the type checker and so there is no need for a dedicated transformation.

\subsubsection{Reusing memory}\label{section:refcount:beans:reuse}

The first step of this process is to insert \textbf{reset}/\textbf{reuse} pairs, \todo{why?}. The approach to inserting such pairs is based on the $R$ algorithm by \citeauthor*{ullrich_counting_2021}~\cite{ullrich_counting_2021}. 
The algorithm will try to insert \textbf{reset}/\textbf{reuse} pairs for the scrutinee of each \textbf{match} expression. 
It does so by going into each branch/arm of the \textbf{match} looking for the first point where the scrutinee is dead, meaning not used in the rest of function body, and then trying to find a \textit{suitable} constructor expression such as $\textbf{ctor}_i \; \bar p$. 

The shape of a constructor is \textit{suitable} when the number of arguments $|\bar p|$ is equal to the number of fields of the scrutinee in the particular branch. Note the distinction between arguments and fields, here an argument is the value given to the application (creation) of the constructor while a field is a place where a value is stored inside of a heap allocated object. 

% The goal of the example is to show the field vs. argument point. When a scrutinee can be reused/when it cannot 
Consider the example program shown in Listing~\ref{listing:refcount:reuse_example}. The \texttt{sync\_fold} programs applies a function $f$ to a \texttt{Sync}, passing the arguments as optional values. 
The code is shown as it would be after inserting \textbf{reset}/\textbf{reuse} pairs.
In the first branch $s$ is a $\texttt{Left}(x)$ value, so it has a single field, making it is possible to reuse $s$ when wrapping $x$ in a \texttt{Just}. 
In the third branch $s$ is a $\texttt{Both}(x,y)$ value with two fields. The algorithm will try to reuse $s$ in a constructor taking exactly two arguments but there is no such expression, so $s$ isn't reused.

\begin{lstlisting}[caption={A fold on sync values. The program is shown using the IR but with a type annotation for readability.}, label={listing:refcount:reuse_example}]
sync_fold f s : (Option 'a -> Option 'b -> 'c) -> Sync 'a 'b -> 'c = 
    case s of
    (let x = proj_0 s in let s' = reset s in
     let x_just = reuse s' in Ctor1(x) in 
     let nothing = Ctor0() in ret (f x_just) nothing)
    (...)
    (let x = proj_0 s in let y = proj_1 s in
     let x_just = Ctor1(x) in let y_just = Ctor1(y) in 
     ret (f x_just) y_just)
\end{lstlisting}

The special-case constructor $\textbf{signal} \; (p,p)$ is treated as a $\textbf{ctor}_i$ with the exception that when matched upon it makes five fields available and similarly it is only suitable when five fields are available from the scrutinee.

%Goal of example: To show reuse on signals and say that we will return to limitations later
Listing~\ref{listing:refcount:reuse_map} shows the map combinator from Section~\ref{section:frp:rizzo} after transformations and inserting \textbf{reset}/\textbf{reuse}. 


\begin{lstlisting}[
  caption={The map combinator from Section~\ref{section:frp:rizzo} after transformations into IR form},
  label={listing:refcount:reuse_map}
]
fun map f s : ('a -> 'b) -> Signal 'a -> Signal 'b =
  case s of  
  (let x = proj_0 s in let xs = Ctor2(s) in
   let s' = reset s in let hd = f x in 
   let map' = pap map(f) in 
   let delayed_map = Ctor0(map') in
   let tl = Ctor6(delayed_map, xs) in
   let res = reuse s' in Signal(hd, tail) in
   ret res)
\end{lstlisting}

Notice, that the algorithm has added instructions for reusing the input signal. 
In here lies a subtle detail, namely that the $xs$ (which represents a $tail$ constructor) needs the original input $s$ to live which makes it problematic to reuse the memory of $s$. We return to this shortcoming later in Section~\ref{section:refcount:beans:counting} when discussing \textbf{inc}/\textbf{dec} instructions. 
For now, the important part is that a $\textbf{reuse} \; x \; \textbf{in ctor}_i \; \bar p$ instruction may fail to reuse $x$ when $x$ is not unique at the time of \textbf{reset}. In map function $s$ cannot be unique at $\textbf{reset} \; s$ because both $xs$ and the caller need $s$.
