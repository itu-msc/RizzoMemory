\section{Automatically reference counting Rizzo}\label{section:refcount}

% \todo{Quick refresh of assumptions from \textit{Reference Counting section}}

The algorithms by \citeauthor{ullrich_counting_2021}~\cite{ullrich_counting_2021} are defined over the calculi $\lambda_{pure}$ and $\lambda_{RC}$, where the $\lambda_{RC}$ extends $\lambda_{pure}$ with reference counting instructions Figure~\ref{figure:refcount_IR} shows the IR used in our compiler, where Rizzo-specific extensions are highlighted in green.  

In this IR, $\texttt{pap}\; c \; \bar p$ denotes partial application of a global constant $c$, $\texttt{ctor}_i \; \bar p$ constructs a value of some erased type with tag $i$, and $\texttt{proj}_i \; x$ projects the $i\text{-th}$ field from constructor value $x$. Fully applied calls and partial applications may only happen on global constants and variable application is defined separately for only a single argument.

\begin{figure}[ht]
  \centering
  \small
  \[

  c &\in& Const\\
  l &\in& \textcolor{greenadditions}{Literal \;::=\; \mathbb{Z} \mid \text{true} \mid \text{false} \mid \text{never} \mid \text{unit} \mid string}\\
  p &\in& \textcolor{greenadditions}{Primitive \;::=\; x \mid l}\\[2pt]
  e &\in&
  \begin{array}[t]{@{}l@{}}
  Expr \;::=\; c\; \textcolor{greenadditions}{\bar p}
  \mid \mathbf{pap}\; c\; \textcolor{greenadditions}{\bar p}
  \mid x \; \textcolor{greenadditions}{p}

Observe that the IR is defined in A-normal form~\cite{flanagan_essence_1993}, where expressions are non-recursive and are only allowed on the right-hand-side of let-bindings. 
Our representation keeps this shape but introduces \textit{Primitives} so that variables and literals (i.e.~integers, booleans, strings, and \texttt{never}) can appear directly in returns and as arguments in function/constructor application. This makes translation easier and avoids bindings such as $\texttt{let} \; x = 20; \; \texttt{ret} \; x$ when $\texttt{ret}\;20$ is enough.
We additionally add a specialised $\texttt{signal}$ constructor that can be used in places where $\texttt{ctor}_i$ is expected. 
This allows runtime to know the fresh memory cell should be stored in the signal heap (see Section~\ref{section:frp:signal_heap}). 
Finally, we add top level values as mappings from constants to $FnBody$ where the $FnBody$ is run at program start up to initialise the value. This way, we can reuse the automatic reference counting of Section~\ref{section:refcount:beans}.

\subsection{Transformations}\label{section:refcount:transforms}

%Goal of section: pretty little flow chart showing how we chain transformations together and an explanation of why/what in relation the assumptions}
To apply automatic reference counting, we first transform the source program into a shape that satisfies the invariants from Section~\ref{sec:background:beans:assumptions}, then lower it to the reference counting IR, and finally run the three algorithms $R$, $collect_{\mathbb{O}}$, and $C$~\cite{ullrich_counting_2021}.

Figure~\ref{figure:refcount:transformations} shows the transformation pipeline. Most transformations are standard compiler steps, but each contributes to the invariants required by the RC insertion algorithms.

\begin{figure}[h]
	\centering
	\includegraphics[width=\textwidth]{flowchart_transformations.png}
	\caption{Compiler passes that a Rizzo program goes through when running the compiler. Transformations inside the green (upper) box receive Rizzo programs as input and in the blue (lower) box they operate on the RC IR.}\label{figure:refcount:transformations}
\end{figure}

The compiler performs consecutive lambda elimination, lambda lifting, copy propagation, alpha-renaming, pattern lowering, dead let elimination, and A-normalization. Consecutive lambda elimination is not strictly required but reduces the number of lambdas to be lifted. 

The lowering of patterns into let-bindings may not directly contribute to any invariant, however it is required since there is no pattern matching in the IR.\@ The result of translating a match such as \lstinline|match mysig with x :: xs -> e| is shown in Listing~\ref{listing:refcount:elim_pattern}. The match expression must be kept around as it is the only point where reset/reuse pairs can be inserted, we will discuss this further in Section~\ref{section:refcount:beans}.

\begin{lstlisting}[caption={Output of lowering a pattern match on a signal value}, label={listing:refcount:elim_pattern}]
match mysig with
| x :: xs -> 
  let x = head mysig in 
  let xs = tail mysig in e
\end{lstlisting}

While some of the invariants are clearly fulfilled by the transformation pipeline, such as the requirement that all lambdas be lifted or that trivial bindings like \lstinline|let x = y| have been eliminated, some of them are not so obvious. 
In particular, the ones about application shape. The RC IR assumes unary variable applications and that applications on global constants are never over-applied. 
Therefore, when \textit{converting to IR} variable applications are rewritten such that \lstinline|x y z| becomes \lstinline|(x y) z| and over-applied global constants are rewritten to a call followed by a series of unary variable applications.

Another non-obvious point is the \textit{eta-expansion} of constructors. This invariant is enforced by the type checker and so there is no need for a dedicated transformation.

\subsubsection{Reusing memory}\label{section:refcount:beans:reuse}

The first step of this process is to insert \textbf{reset}/\textbf{reuse} pairs, \todo{why?}. The approach to inserting such pairs is based on the $R$ algorithm by \citeauthor*{ullrich_counting_2021}~\cite{ullrich_counting_2021}. 
The algorithm will try to insert \textbf{reset}/\textbf{reuse} pairs for the scrutinee of each \textbf{match} expression. 
It does so by going into each branch/arm of the \textbf{match} looking for the first point where the scrutinee is dead, meaning not used in the rest of function body, and then trying to find a \textit{suitable} constructor expression such as $\textbf{ctor}_i \; \bar p$. 

The shape of a constructor is \textit{suitable} when the number of arguments $|\bar p|$ is equal to the number of fields of the scrutinee in the particular branch. Note the distinction between arguments and fields, here an argument is the value given to the application of the constructor (creation) while a field is a place where a value is stored inside of a heap allocated object. 

% The goal of the example is to show the field vs. argument point. When a scrutinee can be reused/when it cannot 
Consider the example program shown in Listing~\ref{listing:refcount:reuse_example}. The \texttt{sync\_fold} programs applies a function $f$ to a \texttt{Sync}, passing the arguments as optional values. 
The program is shown after inserting \textbf{reset}/\textbf{reuse} pairs.
In the first branch $s$ is a $\texttt{Left}(x)$ value with a single field $x$, meaning $s$ can be reused when wrapping $x$ in a \texttt{Just}. 
In the third branch $s$ is a $\texttt{Both}(x,y)$ with two fields. The algorithm will try to reuse $s$ in a constructor taking exactly two arguments but there is no such expression, so $s$ isn't reused.

\begin{lstlisting}[caption={A fold on sync values. The program is shown using the IR but with a type annotation for readability.}, label={listing:refcount:reuse_example}]
sync_fold f s : (Option 'a -> Option 'b -> 'c) -> Sync 'a 'b -> 'c = 
    case s of
    (let x = proj_0 s in let s' = reset s in
     let x_just = reuse s' in Ctor1(x) in 
     let nothing = Ctor0() in ret (f x_just) nothing)
    (...)
    (let x = proj_0 s in let y = proj_1 s in
     let x_just = Ctor1(x) in let y_just = Ctor1(y) in 
     ret (f x_just) y_just)
\end{lstlisting}

The special-case constructor $\textbf{signal} \; (p,p)$ is treated as a $\textbf{ctor}_i$ with the exception that when matched upon it makes five fields available and similarly it is only suitable when five fields are available from the scrutinee.

Listing~\ref{listing:refcount:reuse_map} shows the map combinator from Section~\ref{section:frp:rizzo} after transformations and inserting \textbf{reset}/\textbf{reuse}. 
We see that the algorithm has added instructions for reusing the input signal.


\begin{lstlisting}[
  caption={The map combinator from Section~\ref{section:frp:rizzo} after transformations into IR form},
  label={listing:refcount:reuse_map}
]
fun map f s : ('a -> 'b) -> Signal 'a -> Signal 'b =
  case s of  
  (let x = proj_0 s in let xs = Ctor2(s) in
   let s' = reset s in let hd = f x in 
   let map' = pap map(f) in 
   let delayed_map = Ctor0(map') in
   let tl = Ctor6(delayed_map, xs) in
   let res = reuse s' in Signal(hd, tail) in
   ret res)
\end{lstlisting}
 
However, all is not good with this $map$, since the $xs$ (the constructor $tail \; s$) needs the original input $s$ to live, meaning it would be unsafe to overwrite and reuse the memory of $s$ inside $map$. We will return to this in Section~\ref{section:refcount:beans:counting} when discussing \textbf{inc}/\textbf{dec} instructions. 
For now, recall that a $\textbf{reuse} \; x \; \textbf{in ctor}_i \; \bar p$ instruction may fail to reuse $x$ when $x$ is not unique at the time of \textbf{reset}. In the $map$ function $s$ cannot be unique at the time of $\textbf{reset} \; s$ because $xs$ needs a valid reference to $s$.


\subsubsection{Can I borrow that?}\label{section:refcount:beans:collect}

%GOAL OF SECTION: How do we infer parameter ownerships. We didn't change much.
We distinguish between two kinds of references, $\mathbb{B}\text{orrrowed}$ and $\mathbb{O}\text{wned}$. Owned references participate in reference counting in the usual way, so the number of live references matches the counter stored in the object. Borrowed references, by contrast, are not counted, they are instead assumed to be kept alive by some surrounding owned reference.

%CONSIDER: why is it safe to have the user always assign? If I make everything borrowed, then reset becomes problematic, no?
The distinction exists is to reduce the number of \textbf{inc}/\textbf{dec} instructions generated by the algorithm in Section~\ref{section:refcount:beans:counting}. 
To insert those instructions, we first need \textit{borrow signature} for every function. 
A borrow signature tells us the type of reference that a function expects for each of its parameters. Signatures are captured in a mapping $\delta : Const \to \{\mathbb{O}, \mathbb{B}\}^*$~\cite{ullrich_counting_2021} which maps each function name to a sequence of reference kinds. 
Although these signatures could be declared by programmers, it is more convenient to infer them automatically and \citeauthor*{ullrich_counting_2021} gives a heuristic for doing so. 

%GOAL: explain the heuristic
The heuristic marks a variable $x$ as owned if either $x$ or one of its projections are used in a \textbf{reset}, appears in a partial or variable application, or passed to a function that takes ownership. 
This heuristic fits with our modifications and requires no further changes.
Applying this heuristic recursively over the body a function collects the set of all variables that must be owned. Once this set is computed, any parameter in the set is marked as owned in the function signature and the rest are marked as borrowed.

%GOAL: quickly explain our call-graph approach, the recursive function problem
However, the borrow signature of a function depends on the signatures of the functions it calls. Recursive functions are also problematic, since their final signatures may depend on their own body.
To handle this, we build a call graph, that records for each function $f$, the set of functions that call $f$.
We then iterate over the functions, collecting owned variables and updating signatures until we reach a fixed point where no signature changes any further.

\subsubsection{Learning to count}\label{section:refcount:beans:counting}

%GOAL OF THE SECTION: Our adaptation of the immutable bean algorithm. Our modifications. How do we insert these refcount-instructions}

%GOAL: How do we insert these refcount-instructions
The algorithm for inserting \textbf{inc}/\textbf{dec} instructions is based on algorithm $C$ by \citeauthor*{ullrich_counting_2021}~\cite{ullrich_counting_2021}.


% The algorithm is given a map $\beta_C$ that records the ownership of global function parameters, and it also maintains a local map $\beta$ for the ownership of variables within function bodies.

The insertion of \textbf{inc}/\textbf{dec} operations depends on the ownership of a variable and the ownership expected by its use site, which we call the context.
Variables used in a \textit{borrowed context} do not need to be incremented.
By contrast, variables used in an \textit{owned context} should be incremented before use, because the context consumes an RC token.
Examples of owned contexts are constructor applications, variable applications, partial applications and calls to functions that expect ownership of a particular argument.
Borrowed variables are never decremented while owned variables should be decremented after their last use. 
The algorithm goes one step further and will skip incrementing owned variables on their last use in owned contexts. This optimisation avoids unnecessary \textbf{inc}/\textbf{dec} pairs and leaves more opportunities for \textbf{reset}/\textbf{reuse} to succeed.
%an increment before would always ruin the possibility of reset/reuse, $count(x)+1$, then call $f \; x$ all then again $count(x)-1$ which may erase $x$.

%GOAL: Our adaptations - primitives, (again) signals are just special case constructors
Adapting algorithm $C$ to our IR, from Figure~\ref{figure:refcount_IR}, requires handling the \textbf{signal} constructor, primitives, and global variables. 
The special case constructor \textbf{signal} can be treated \textbf{signal} as any other \textbf{ctor} value, where all arguments are considered in an owned context.
Primitives are either variables, which are handled according to the ownership rules above, or literals, which are never reference counted.
Global variables are treated as borrowed for the purpose of \textbf{inc}/\textbf{dec} insertion, since they are assumed to be kept alive for the duration of the program. This choice keeps the implementation simple and safe but may introduce unnecessary \textbf{inc} instructions.

%GOAL: Circle back to the map example - TRY to explain why we can never reuse and why that is CORRECT with regards to the semantics of Rizzo.
Returning to the map example from earlier Listing~\ref{listing:refcount:incdec_map} shows final compiled version of the map combinator.
The listing additionally shows the result of borrow signature inference where both $f$ and $x$ are annotated as owned. We show the  reference counts of the variables $x$, $s$, and $f$ assuming they are all unique when map is called. 
Notice, that because $xs$ is a $tail s$ constructor it needs $s$ to live for its duration and as such we must increment $s$ before it can be used again later in the $\textbf{reset} \; s$. 
This increment sets the reference count of $s$ to two, meaning $s$ can never be unique at the time of $\textbf{reset} \; s$.

\begin{lstlisting}[
  caption={The map combinator with \textbf{inc}/\textbf{dec} instructions. The inferred ownership annotations for the arguments of map are shown on the first line.},
  label={listing:refcount:incdec_map}
]
f: Owned, s: Owned
fun map f s : ('a -> 'b) -> Signal 'a -> Signal 'b =
  case s of  
  (let x = proj_0 s in        count(x) = 1
   inc x;                     count(x) = 2
   inc s;                     count(s) = 2
   let xs = Ctor2(s) in
   let s' = reset s in        count(s) = 1, s' = NULL
   inc f;                     count(f) = 2
   let hd = f x in
   let map' = pap map(f) in
   let delayed_map = Ctor0(map') in
   let tl = Ctor6(delayed_map, xs) in    
   let res = reuse s' in Signal(hd, tail) in
   ret res)
\end{lstlisting}
