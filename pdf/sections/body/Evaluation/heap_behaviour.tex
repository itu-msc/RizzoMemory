In order to evaluate the heap behaviour, we will be running a small Rizzo program. The program has two modes, an English mode and a Danish mode, and will echo back any input given to it in the respective language. 
We have built it such that the size of the heap changes, so we can show that the heap grows and shrinks in a predictable way as the program runs.
The source can be found in Appendix \todo{X}, but the relevant part of the code is as follows:

When compiling the program we enable the \texttt{--debug-info} flag, which enables the logging of debug information. Specifically, the flag adds unique identifiers to signals and enables printing of the signal heap after each time step.
We compiled the program with \texttt{rizzoc examples/eval\_small.rizz --debug-info} and then run the generated executable \texttt{./output(.exe)}.
Running the program we get the output in Figure \ref{fig:heap_print}. We have colored the user input in blue and the output from the program in green. The lines starting with `Sig index' are the debug prints of the heap, which show the next signal index to be used, the size of the heap, and the unique identifiers of the signals in the signal heap after the time step is done. Each signal that was updated in the time step is printed in brackets.

On initialisation, the heap contains four stable signals:

\begin{itemize}
    \item [0] is the internal signal created and watched inside the filter\_l function. 
    \item [1] is the signal created and watched inside the filter\_map\_l function.
    \item [2] is the signal we build for the variable named default.
    \item [3] is the output from the switch function.
\end{itemize}

The program starts in English mode, and the first two lines are the initial greeting and the instruction for switching to Danish. This output is caused by the signal having been given an initial head value in the code, which triggers the output function to print the greeting and the instruction. 
We then input `\textit{input 1}' and `\textit{input 2}', which are regular inputs that will be printed back to us. Which as expected both trigger the same signals and keep the heap size constant after the step has finished.
We then switch to Danish mode by inputting `\textit{da}', which causes the signal returned by \texttt{switch\_r} to change, and thus the heap grows as the delayed computations are evaluated. The heap grows by two signals, which is the result of a
