\subsubsection{Semantics: Rizzo}

In Rizzo~\cite{rizzo_modal_2025} the operational semantics are split in two: The evaluation semantics for expressions, and the reactive semantics that describe how Rizzo programs react when time is stepped forward.

The evaluation semantics follow a standard eager evaluation strategy. The only exception being $\text{delay} \; e$ for which the argument $e$ is lazily evaluated.
% We have slightly different evaluation semantics because we allow general recursion.
We change the semantics presented by \citeauthor*{rizzo_modal_2025}~\cite{rizzo_modal_2025} \todo{only a little}. We exclude the $\textbf{fix}\text{(-point)}$ construct which is used to guarantee productivity by guarding recursion with the \texttt{Delayed} type. Meaning, our implementation makes no guarantees about productivity, a trade-off we are willing to make given the usefulness of general recursion in functional programming.

%Trying to say something like: Rizzo~\cite splits the reactive semantics in three: *which? What is their purpose*. We'll point our focus to the update semantics, since this is the most important for the problem at hand
The reactive semantics define the reactive behaviour of Rizzo, i.e.\ what happens when any channel has produced a value and time is moved forward. 
These semantics are split in three parts:
The \textit{advance semantics} which describe how a $\texttt{Later}\, A$ value is advanced to produce a value of type $A$; 
the \textit{update semantics} which, using the advance semantics, describe how a signal is updated and brought forward in time;
and the \text{step semantics} which describe the process of applying the update semantics to every signal when time is \textit{stepped} forward.

%Goal: Goal is to explain the update semantics. Explain the now vs. earlier heap.
For the purpose of memory management, we will focus our attention on the update semantics but first we must take detour to understand the conceptual structure of the heap.
In the typical way the heap holds onto all dynamically allocated constructs such as tuples, signals, function closures, etc. 
However, in Rizzo the sub-heap storing only signals is especially important since it must be iterated as part of the reactive semantics. 
We call this sub-heap the `signal heap' and it is further sub-divided in two parts: \texttt{now heap} holding only signals that belong to the current time step; and the \texttt{earlier heap} which hold all the stale signals to be updated.

%Goal: explain update semantics
At runtime when some channel $\mathbf{C}$ produces a value, the step semantics describe that all signals must be moved to the \texttt{earlier heap}, consequently marking their data stale.
Then as described by the update semantics, each signal is moved one-by-one from the \texttt{earlier heap} to the \texttt{now heap}. To move a signal we first check if its tail was dependent on $\mathbf{C}$ and if it were, then the tail is \textit{advanced} to obtain a new signal whose data is copied into the original signal which can then be moved to the \texttt{now heap}. Otherwise, the signal is moved to the \texttt{now heap} unchanged.
%We avoid talking about the updated flag - that is an implementation detail. Belongs to advance

%Goal: Here comes a digest, what does all of the above mean? 
However, since chains of signals built using combinators form data-dependencies the move-ordering is significant. 
Boiled down the reactive semantics require that:
\begin{itemize}
    \item Any new signal must only depend on other signals in the \texttt{Now heap}.
    % \item When an input is produced on a channel, all signals with a tail that depends on the clock of that channel must to be updated.
    \item Signals must be updated based on a topological order of their dependencies. If signal B depends on signal A, signal A must appear before signal B in the heap and thus be updated first.
    % \item Evaluating the tail of a signal produces a new signal, which must be copied back into the original signal. %This means that the original signal is updated in place, and any references to it remain valid.
    \item If no reference exists to the intermediate signal produced by evaluating the tail, then the intermediate signal must be deallocated to prevent duplicate work.
    \item New signals must be inserted into the \texttt{Now heap}, to prevent it from being evaluated in the current time step.
    % \item 
\end{itemize}

%Goal: Okay, so now we know what is required of the heap structure. How do we represent it? -> Linked list, advantages thereof.

Based on these requirements, we have chosen to structure our heap as a linked list. Using a linked list has the advantage of:
% This gives us a simple way to maintain the topological order of signals, it allows for efficient insertion and removal of signals, 
%  as we can ensure that when a new signal is created it is added after the split between the \texttt{Now} and \texttt{Earlier} heaps. \dots

\begin{itemize}
    \item Representing the \texttt{Now} and \texttt{Earlier} heaps in one data structure, and expressing a split between them using a pointer (cursor).
    \item Moving signals from the \texttt{Now} heap to the \texttt{Earlier} heap in O(1) time by resetting the cursor to the start of the heap. Likewise moving signals back to the \texttt{Now} heap by moving the cursor forward.
    \item O(1) insertion that maintaining the topological order of signals, by adding them in the end of the \texttt{Now} heap, we ensure that any signal can only depend on signals that were created before it.
    \item O(1) removal of signals when their reference count drops to zero.
    \item Stable references to signals, as insertions and removals do not move other signals in memory, so pointers to signals remain valid throughout their lifetime.
\end{itemize}

The evaluation semantics of Rizzo~\cite{rizzo_modal_2025} and the operational semantics presented by \citeauthor*{ullrich_counting_2021}~\cite{ullrich_counting_2021} both built on a shared abstraction, an eagerly evaluated lambda calculus.
This means, we are able to reuse the semantics defined by \citeauthor*{ullrich_counting_2021}~\cite{ullrich_counting_2021} for both reference counting and the \todo{core lambda calculus constructs}.

In the semantics presented by \citeauthor*{ullrich_counting_2021}~\cite{ullrich_counting_2021} there are only two kinds of heap allocated values: function closures (partial applications); and constructor values of the shape $\textbf{ctor}_i$ which represent a value created from the $i\text{-it}$ constructor of some tagged union.
It turns out, that these two types of values are enough to model all constructors of Rizzo except signals.

Values of the \texttt{Sync}, \texttt{Later}, and \texttt{Delayed} types are all modelled as regular tagged unions and each constructor is given a different tag, e.g. $0$ for $\mathsf{never}$, $1$ for $\mathsf{wait}$, $0$ for $\mathsf{delay}$ etc.

Although signals are just containers of values like other constructors, they must be treated differently since their creation and erasure modify the signal heap.
That is, when a signal is created the signal heap is modified in order to the insert the signal and similarly when a signal is erased, references must updated to avoid breaking the linked list.
%Do we mention that a location is returned? Probably not
Therefore, we define a specialised \textbf{signal} value. The semantics of this value follow that of regular constructors, where each argument is evaluated, stored into a fresh memory location, and the new memory location is given a reference count of $1$. 
The signal is then inserted into the signal heap at the position just before the heap cursor. 
When the reference count of the signal reaches $0$, the count of all its fields are decremented by $1$ and the signal is removed from the signal heap.
