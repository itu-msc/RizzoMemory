After that, we describe the operational semantics of Rizzo, the constraints these semantics place on the choice of heap representation, and finally how we merge these semantics with the reference-counting semantics of \citeauthor*{ullrich_counting_2021}~\cite{ullrich_counting_2021}

\subsection{Syntax}

We have extended the base ML-like syntax with constructs specific to Rizzo, such as the signal constructor (\texttt{::}) and the \texttt{Later}/\texttt{Delayed} constructors. The syntax supports top-level definitions, local let bindings, lambda expressions, pattern matching, and type annotations.

\subsubsection{Top-level Declarations}

Programs in Rizzo consist of a sequence of top-level declarations. There are three forms: value bindings, function definitions, and type definitions.

\begin{verbatim}
let x = e                            (* value binding *)
fun f x1 x2 ... xn = e               (* function definition *)
type TypeName x1 ... xn = 
| C (t1, ..., tn) ...                (* type definition *)
\end{verbatim}

The function definition \texttt{fun f x1 ... xn = e} is syntactic sugar for \texttt{let f = fun x1 xn -> e}. Functions marked with the \texttt{@effectful} annotation are allowed to perform side effects such as registering output signals. This annotation also prevents the compiler transformations of Section~\ref{section:refcount:transforms} from removing calls to such functions.

\subsubsection{Expressions}

The core expression forms are:

\begin{verbatim}
e ::= x                           (* variable *)
    | c                           (* constants *)
    | fun p -> e                  (* lambda abstraction *)
    | e1 e2                       (* application *)
    | C (e1, ..., en)             (* constructor application *)
    | let x = e1 in e2            (* local binding *)
    | (e1, e2)                    (* tuple *)
    | e1 :: e2                    (* signal/list constructor *)
    | e1 |> e2                    (* pipe operator *)
    | match e with | p1 -> e1 ... (* pattern matching *)
    | if e1 then e2 else e3       (* conditional *)
    | (e : t)                     (* type annotation *)
    | delay e | ostar e1 e2       (* Delayed constructors *)
    | wait e | watch e | tail e 
    | sync e1 e2 | laterapp e1 e2 (* Later constructors *) 
\end{verbatim}

Constants can be booleans, either $\mathsf{false}$ or $\mathsf{true}$, integers, strings, and the $\mathsf{never}$ constructor of the \texttt{Later} type.
The pipe operator \texttt{|>} is left-associative and de-sugars to instance of the later application as such \verb|laterapp (delay e1) e2|.
The signal constructor \texttt{::} is right-associative and constructs a signal from a head value and a tail computation. For example, $5 :: \mathsf{never}$ creates a constant signal with value 5.

\subsubsection{Patterns}

For patterns we have included the necessary forms to support matching against tuples, signals, and constructors:

\begin{verbatim}
p ::= _                           (* wildcard *)
    | x                           (* variable binding *)
    | c                           (* constant pattern *)
    | (p1, p2)                    (* tuple pattern *)
    | p1 :: p2                    (* signal/cons pattern *)
    | c :: s                      (* string pattern *)
    | C (p1, ..., pn)             (* constructor pattern *)
\end{verbatim}

Like other head and tail structured datatypes, we have chosen to use $x :: xs$ pattern for signals, to bind the head of the signal to $x$ and the tail (a \texttt{Later} computation) to $xs$.
We also added it for strings, to allow the head character to be bound separately from the tail string. This is useful for functions that need to process strings such as when parsing user input.
