\subsection{Functional Reactive Programming}\label{section:frp:frp_intro}

Reactive programs are interactive programs that communicate with the environment through inputs and outputs. Examples of such programs include servers, graphical user interfaces, and games.

Functional Reactive Programming (FRP) is a programming paradigm that combines functional programming with reactive programming. It allows developers to create programs that can react to changes in data over time, making it easier to handle asynchronous events and manage state in a declarative way. 

The main abstraction of FRP is the \textit{signal}, which models time-varying values. Conceptually, a signal is a sequence of values produced by interacting with the environment. Signals can be transformed, combined and manipulated using familiar functional techniques.

%Problems of FRP
However, representing signals as infinite streams makes it easy to write declarative programs that cannot be run effectively. Such programs may violate one of the three main properties \textit{causality} (computations are not dependent on the results of future computations), \textit{productivity} (computations eventually produce an output), and the \textit{absence of space leaks}. 

To overcome these issues, there exists a family of FRP languages~\cite{neel_krishnaswami_2013,bahr_asynchronous_2023,rizzo_modal_2025} which use \textit{modal types} to guarantee these properties. In the literature~\cite{bahr_asynchronous_2023,rizzo_modal_2025}, these modal types are the `later' (written $\bigcirc a$) and the `box' (written $\square a$) type constructors. The type $\bigcirc a$ expresses a promise that a value of type $a$ will become available at some later time step, whereas the $\square a$ describes a value of type $a$ that is \textit{time-independent} and as result safe to keep across time steps.

Given these types it is possible to define a signal as a tuple of a current value (head) and a delayed computation (tail) for computing future values: $\text{type} \; \texttt{Signal }a = a \times \texttt{Later }(\texttt{Signal }a)$.

\subsection{Rizzo}\label{section:frp:rizzo}

Rizzo~\cite{rizzo_modal_2025} is an asynchronous FRP language designed to provide the same properties but with a simpler type system. Rizzo changes the semantics of signals by defining them as mutable references:

\[\text{type} \texttt{ Signal }a = \texttt{Ref }(a \times \texttt{Later }(\texttt{Signal }a))\]

Signals are updated in place whenever the environment produces a value that is relevant for the signal. These updates are done, not by the programmer, but by the reactive semantics of Rizzo, see Section~\ref{section:frp:signal_heap}. For convenience we define the signal constructor as an infix operator: 
\[\_ :: \_ : a \rightarrow \texttt{Later }(\texttt{Signal }a) \rightarrow \texttt{Signal }a\]

Rizzo, like other languages~\cite{bahr_asynchronous_2023}, differentiates between two variants of the `later' type. 
In Rizzo there is the $\texttt{Later } a$ which describes a value of type $a$ becoming available at some future time step. The ``some'' here is existential; we do not know exactly when the value will arrive, only that it may eventually arrive.
And then there is the $\texttt{Delayed }a$ which describe a value of type $a$ that is available at \emph{all} (universally quantified) future time steps. This universally quantified nature makes \texttt{Delayed} values safe to use across time boundaries.

Rizzo has a number of constructors for working with delayed computations~\cite{rizzo_modal_2025}. We shall present some of them here but for a full overview see Section~\ref{section:rizzolang:types}.
Starting with the $\texttt{Delayed}$ type, its main constructor is $\mathsf{delay} : a \rightarrow \texttt{Delayed }a$ which simply pushes any value into the future. 
Then for the $\texttt{Later}$ type, we may use the constructor $\mathsf{wait} : \texttt{Chan}\;a \rightarrow \texttt{Later}\;a$, which is used to wait for the environment to produce input on a particular channel. One can think of a channel ($\texttt{Chan}$ type) as an opening through which the environment can pass values to Rizzo programs.
Another later constructor is the $\mathsf{tail} : \texttt{Signal}\;a \rightarrow \texttt{Later }(\texttt{Signal}\;a)$, which lets us access the delayed computation for the future value of a signal. 
However, probably the most useful of all is the applicative action on $\texttt{Later}$ computations:

\[\mathsf{laterapp} \; f \; x : \texttt{Delayed }(a \rightarrow b) \rightarrow \texttt{Later}\;a \rightarrow \texttt{Later}\;b\]

Intuitively, $\mathsf{laterapp}$ delays the application of a function until the argument $x$ is available. This operator allows us to apply delayed functions to later values. However, when writing Rizzo programs it is often more convenient to use the functorial action or pipe operator:
\begin{align*}
\_ \; &\triangleright \; \_ \; : (a \rightarrow b) \rightarrow \texttt{Later} \; a \rightarrow \texttt{Later} \; b \\
f &\triangleright x = (\mathsf{delay} \; f) \triangleright x   
\end{align*}

With these constructors we already have the necessary tools to define some signal combinators! We first define the $\mathsf{mk\_sig}$ that lets us create a signal from any delayed computation:
\begin{align*}
    &mk\_sig : \texttt{Later }a \rightarrow \texttt{Later }(\texttt{Signal }a) \\
    &mk\_sig \; d = (\lambda x. x :: mk\_sig \; d) \triangleright d
\end{align*}

Assuming there exists a channel $console : \texttt{Chan}\;\texttt{String}$ that reads from the console, we can define signals that react to console input. An example of such a signal is $(\text{""} :: mk\_sig \; (\mathsf{wait} \; console))$, which echoes strings read from the console.

Let us take it one step further and define the familiar $map$ function.
\begin{align*}
    &map : (a \rightarrow b) \rightarrow \texttt{Signal }a \rightarrow \texttt{Signal }b \\
    &map \; f \; (x :: xs) = f \; x :: (map \; f) \fpipe xs
\end{align*}

In our definition of $map$ we make use of pattern matching on signals. We could alternatively used explicit $\mathsf{head}$ and $\mathsf{tail}$ operations to extract $x$ and $xs$ respectively.

Let us return to what makes Rizzo different, its semantics of signals. 
Because signals are updated in place as mutable references rather than stored as immutable sequences, it becomes possible to define combinators such as $sample$:
\begin{align*}
&sample : \texttt{Signal }a \rightarrow \texttt{Signal }b \rightarrow \texttt{Signal } (a \times b) \\
&sample \; xs \; ys = map \; (\lambda x. (x, \; \mathsf{head} \; ys)) \; xs  
\end{align*}

The $sample$ combinator samples the current value of $ys$ each time a new value is produced on the signal $xs$. 
If signals were immutable, the combinator would instead reuse the first value of $ys$ for every subsequent value of $xs$, which would force the runtime to retain the entire history of $ys$ and would therefore introduce a space leak.

\subsubsection{Signals and the heap}\label{section:frp:signal_heap}

%Goal of section: A very high level explain what happens when time is progressed in Rizzo. Iterate through all signals to update them. This creates intermediate signals, think mk_sig which calls itself recursively, producing a new signal each time which is only used to update the original in place! If these intermediate signals still exist by the time the next value is produced, we are doing unnecessary work because the intermediate signal isn't used anymore...
% This is what we need precise clean-up for, and that is why reference counting. 

In Rizzo, time steps forward when a value is produced by any channel. This could happen because of a network message, a sensor reading, or user input from the console. 
Time steps are discrete and sequential, meaning they happen one-after-the-other and never at the same time. 

When time steps forward, the data stored in every signal becomes stale since it now belonging to the past. The runtime must therefore bring all signals into the new step. It does so by iterating over the heap, evaluating each signal's tail to obtain a potentially fresh intermediate signal, and writing the resulting data back into the original signal.
Without the proper clean-up, this process leaves both the intermediate signal and the original signal alive, both of which must then be brought into future time steps, leading to duplicate work.

Reference counting provides the precise clean-up needed to avoid this problem. It allows the runtime to detect when an intermediate signal is no longer referenced and can be immediately discarded, thus avoiding unnecessary work during future heap updates.
